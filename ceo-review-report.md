## Step 0: Nuclear Scope Challenge

### 0A. Premise Challenge

**Is this the right problem to solve?**

The problem is NOT "build a great RPG." You've already done that. 110 models, 15 scenes, 1,596 assets, 212 ending test cases, P2W enforcement at the code level, WCAG 2.1 AA accessibility. The engineering is real.

The problem is: **you have a product nobody has used.**

52 development phases. Zero players. The gap isn't quality, it's *contact with reality*. Every hour spent on Phase 53 is an hour not spent learning whether real humans enjoy what you built.

**What happens if you do nothing?**

You keep polishing. Phase 53, 54, 55. More documents, more tests, more features. The game gets better on paper. Meanwhile, the market moves. Other Korean indie RPGs ship. Your 742 design documents become increasingly disconnected from what actual players want. The asymptotic pattern continues.

This is a real pain point, not hypothetical. You've been building for 52 phases without a single data point from a real player.

**What is the actual user outcome?**

A Korean RPG player opens a URL, plays Chapter 1 for 2 hours, and either comes back tomorrow or doesn't. Everything else is in service of that moment.

## Section 1: Architecture Review

### System Architecture

```
                                    ┌─────────────────┐
                                    │   CDN (future)   │
                                    └────────┬────────┘
                                             │
┌──────────────────┐                ┌────────▼────────┐
│   Browser        │  HTTP/WS      │   Nginx (Docker) │
│   Phaser 3       │◄────────────►│   Port 80        │
│   10 Scenes      │                └────────┬────────┘
│   32 UI Comps    │                         │
│   Protobuf       │                ┌────────▼────────┐
└──────────────────┘                │   Fastify Server │
                                    │   Port 3000      │
                                    │   40+ REST routes │
                                    │   14 WS handlers  │
                                    │   Tick Manager    │
                                    └───┬──────────┬───┘
                                        │          │
                               ┌────────▼──┐  ┌───▼────────┐
                               │ PostgreSQL │  │   Redis 7   │
                               │   16       │  │   Cache     │
                               │ 110 models │  │   Sessions  │
                               │ 2,026 LOC  │  │   Rate Lim  │
                               └────────────┘  └────────────┘
```

**Strengths:**
- Clean monolith with DI container. Right choice for a solo dev — no microservice overhead.
- Declarative route registry (`featureRegistry.ts`) — plug/unplug systems via feature flags.
- Protobuf for socket messages — binary efficiency over JSON.
- Proper graceful shutdown (`shutdownManager.ts`) — drain connections, flush Redis, close DB pools.

**Concerns:**

1. **Single Postgres node.** No HA. If the pod crashes, the game is down until manual restart. Acceptable for demo/CBT. Needs PG failover before OBT (2,000+ users).

2. **Redis single instance.** Same concern. LRU eviction is fine for cache, but session data loss on Redis restart means all players get kicked. For demo: acceptable. For production: needs persistence or Redis Cluster.

3. **combatSocketHandler.ts (253 lines).** Office-hours report cited 8,042 lines — that appears to be a stale measurement or different file. Current file is manageable.

4. **BattleScene.ts (1,476 lines).** Large but functional. Not a shipping blocker. Refactor when you need to add new combat features, not before.

**Rollback posture:** Git revert. No database migrations in the deploy path (Prisma migrations are separate). Feature flags provide kill switches. Rollback time: <5 minutes.

**Scaling characteristics:**
- 10x load (50 concurrent): Fine. Single Fastify process handles this easily.
- 100x load (500 concurrent): Redis pub/sub and PostgreSQL connection pool become bottlenecks. K8s HPA helps but needs load testing.
- 1,000x load (5,000 concurrent): Needs Redis Cluster, PG read replicas, and socket sharding. Not relevant until OBT.

**No issues requiring user decision. Moving on.**

## Section 2: Error & Rescue Map

### Critical Payment Flow

```
METHOD/CODEPATH                | WHAT CAN GO WRONG             | EXCEPTION CLASS
-------------------------------|-------------------------------|------------------
paymentRoutes/create-checkout  | Stripe API timeout            | StripeTimeoutError
                               | Invalid crystal tier           | ValidationError
                               | User not authenticated         | AuthError
                               | Duplicate session creation     | IdempotencyError
                               |                               |
paymentRoutes/webhook          | Invalid Stripe signature       | WebhookVerificationError
                               | Event already processed        | (Idempotency check)
                               | Crystal grant fails (DB error) | PrismaClientError
                               | User not found for charge      | RecordNotFoundError
                               |                               |
stripeManager/grantCrystals    | DB transaction fails           | PrismaTransactionError
                               | Race: double credit            | (Idempotency key check)
```

```
EXCEPTION CLASS              | RESCUED? | RESCUE ACTION            | USER SEES
-----------------------------|----------|--------------------------|------------------
StripeTimeoutError           | Y        | Return 503, log          | "Payment unavailable, try again"
ValidationError              | Y        | Return 400               | "Invalid package"
WebhookVerificationError     | Y        | Return 400, log warning  | Nothing (async)
PrismaTransactionError       | ?        | Unknown — need to verify | Could be silent failure ← GAP
RecordNotFoundError          | ?        | Unknown — need to verify | Could lose payment ← GAP
```

**GAP 1: Stripe webhook → crystal grant failure.** If the DB transaction fails after Stripe charges the user, the user loses money with no crystals. Need: retry queue + manual reconciliation endpoint + alert. Priority: P1 (before activating real payments).

**GAP 2: Webhook user-not-found.** If user deletes account between payment and webhook, the crystal grant targets a nonexistent user. Need: log + admin alert + refund API call. Priority: P2 (edge case but real money).

### Combat Flow

```
METHOD/CODEPATH                | WHAT CAN GO WRONG             | EXCEPTION CLASS
-------------------------------|-------------------------------|------------------
combatEngine/processAction     | Skill on cooldown              | CooldownError (handled)
                               | Insufficient mana              | ValidationError (handled)
                               | Target already dead            | InvalidTargetError (handled)
                               | Socket disconnect mid-combat   | ConnectionError
                               |                               |
tickManager/broadcast          | Redis pub/sub failure          | RedisConnectionError
                               | Client desync                  | (No explicit error)
```

**GAP 3: Socket disconnect mid-combat.** What happens to the combat instance? Does it pause, timeout, or continue without the player? Need: explicit reconnection window (30s) with combat freeze, or auto-complete with AI taking over. Priority: P1 (common in browser games — tab close, network flap).

**GAP 4: Client desync.** If tick broadcasts fail, the client state diverges from server. Need: periodic state snapshot reconciliation (every 30s) beyond individual tick updates. Priority: P2 (occurs under load, not at demo scale).
