## 1. 테스트 피라미드

### 1.1 계층 구조

```
          ╱ ╲
         ╱ E2E ╲         수동 탐색 + 자동 시나리오 (5%)
        ╱───────╲
       ╱ 통합 테스트 ╲    API + Socket + DB 통합 (20%)
      ╱─────────────╲
     ╱   유닛 테스트    ╲  함수/클래스 단위 (75%)
    ╱───────────────────╲
```

### 1.2 계층별 상세

| 계층 | 범위 | 도구 | 실행 주기 | 목표 커버리지 |
|------|------|------|---------|------------|
| 유닛 | 개별 함수, 클래스, 유틸리티 | Jest (서버), Vitest (웹 클라이언트), NUnit (Unity), Catch2 (UE5) | 커밋마다 (CI) | ≥ 80% |
| 통합 | API 엔드포인트, Socket 이벤트, DB 트랜잭션 | Supertest + Jest, Testcontainers (PostgreSQL/Redis) | PR마다 (CI) | ≥ 70% |
| E2E | 전체 유저 시나리오, 크로스 브라우저 | Playwright (웹), Unity Test Runner, RunUAT (UE5) | 일일 1회 + 릴리즈 전 | 핵심 시나리오 100% |

### 1.3 테스트 대상 우선순위

| 우선순위 | 대상 시스템 | 이유 |
|---------|-----------|------|
| P0 (필수) | 인증/세션, 저장/로드, 결제, 인벤토리 거래 | 데이터 무결성·보안 직결 |
| P1 (높음) | 전투 계산, 스킬 체인, 콤보 시스템, 엔딩 분기 | 핵심 게임플레이 |
| P2 (보통) | 퀘스트 트리거, NPC 대화, UI 렌더링 | 콘텐츠 완성도 |
| P3 (낮음) | 코스메틱 표시, 사운드 재생, 애니메이션 | 체감 품질 |

## 3. 자동화 전략

### 3.1 엔딩 회귀 테스트

멀티 엔딩 4종의 분기 로직은 가장 높은 회귀 리스크를 보유. 자동화 필수.

```
엔딩 회귀 테스트 시나리오:

1. 엔딩 A (최선): fragment_count=4, party_alive=63, ending_a_score=60, lethe_understood=true
   → 검증: 엔딩 A 트리거, 레테 안식 씬, 세계 치유 에필로그

2. 엔딩 B (정통): fragment_count=3, 나머지 조건 미충족
   → 검증: 엔딩 B 기본 트리거, 기억 희생 씬, 새 이름 에필로그

3. 엔딩 C (비극): betrayal_score=70+
   → 검증: 엔딩 C 트리거, 선택적 망각 씬, 포스트 크레딧 망각 씬

4. 엔딩 D (히든): sacred_artifacts=12, sealed_ruins_visited=6
   → 검증: 엔딩 D 트리거 (D→C→A→B 판정 우선순위), 열두 신 해방 씬

5. 경계값 테스트:
   - fragment_count=2 → 엔딩 B (파편 부족)
   - betrayal_score=69 → 엔딩 C 미트리거
   - sacred_artifacts=11 → 엔딩 D 미트리거
   - 복합 조건: D 충족 + C 충족 → D 우선 (판정 순서 검증)
```

**실행:** 서버 SaveLoad API를 통해 플래그 직접 주입 → 챕터 5 최종 판정 로직 실행 → 엔딩 ID 검증
**주기:** PR마다 (변경 감지 시) + 일일 1회 전수

### 3.2 SaveLoad 무결성 테스트

```
테스트 매트릭스:

1. 기본 저장/로드 사이클
   - 저장 → 로드 → 전체 상태 비교 (JSON diff)
   - 검증 항목: 캐릭터 스탯, 인벤토리, 퀘스트 진행도, 스토리 플래그, 위치

2. 중단 복구
   - 저장 중 강제 종료 → 재시작 → 마지막 유효 저장 복구 확인
   - WebSocket 끊김 중 저장 시도 → 재연결 후 저장 완료 확인

3. 버전 호환성
   - v1.0 세이브 → v1.1 서버에서 로드 (스키마 마이그레이션)
   - 누락 필드 기본값 채움 검증

4. 동시성
   - 동일 계정 2세션 동시 저장 → 마지막 저장 우선, 데이터 손실 없음

5. 데이터 크기
   - 인벤토리 120칸 만재 + 퀘스트 100개 완료 + 엔딩 플래그 전수 → 저장/로드 시간 < 3초
```

**실행:** Testcontainers(PostgreSQL + Redis) 기반 격리 환경
**주기:** 커밋마다 (CI)

### 3.3 L10N 정합성 테스트

```
L10N 키 정합성 러너 (tools/regression 참조):

1. 키 누락 검사
   - 기준 언어(ko) 키 목록 추출 → 각 번역 파일(en, ja, zh-Hans, zh-Hant) 대조
   - 누락 키 리스트 리포트

2. 미사용 키 검사
   - 코드베이스에서 참조되지 않는 L10N 키 탐지 → 정리 후보 리포트

3. 변수 정합성
   - `{player_name}`, `{item_count}` 등 인라인 변수가 모든 언어에 동일하게 존재하는지 검증

4. 텍스트 길이 검증
   - 각 언어별 번역 텍스트 길이가 UI 최대 너비 초과 여부 확인
   - 경고 기준: 기준 언어 대비 150% 초과 (영어→독일어 등 확장률 큰 경우)

5. 인코딩 검증
   - UTF-8 BOM 없음 확인
   - CJK 문자 렌더링 폴백 확인 (WebFont/TextMeshPro/Slate)
```

**실행:** GitHub Actions PR 체크
**주기:** L10N 파일 변경 시 자동 + 주 1회 전수

### 3.4 CI/CD 파이프라인 통합

```
GitHub Actions 워크플로우:

PR 생성 시:
  1. TypeScript 린트 (ESLint)
  2. 유닛 테스트 (Jest/Vitest) — 커버리지 리포트
  3. 통합 테스트 (Supertest + Testcontainers)
  4. L10N 정합성 검사
  5. 엔딩 회귀 테스트 (플래그 변경 감지 시)

Main 머지 시:
  6. 전체 E2E 테스트 (Playwright)
  7. SaveLoad 무결성 테스트
  8. 성능 벤치마크 (Lighthouse + 커스텀)
  9. Client 빌드 → S3 업로드 / Docker 이미지 → ECR
  10. 스테이징 배포 → 스모크 테스트 → 프로덕션 롤링 업데이트

일일 (Nightly):
  11. 엔딩 회귀 전수
  12. 아이템 강화 확률 시뮬레이션 (10만회)
  13. 몬스터 드롭 테이블 시뮬레이션
```

## 4. 버그 심각도 분류

### 4.1 심각도 등급

| 등급 | 명칭 | 정의 | 대응 시한 | 예시 |
|------|------|------|---------|------|
| S1 | Critical | 게임 불가, 데이터 손실, 보안 침해 | 즉시 (핫픽스 4시간 내) | 저장 데이터 소실, 결제 오류, 인증 우회, 서버 크래시 |
| S2 | Major | 핵심 기능 장애, 우회 가능하나 심각한 영향 | 24시간 내 패치 | 전투 데미지 계산 오류, 엔딩 분기 오작동, 인벤토리 아이템 복제, 스킬 무한 사용 |
| S3 | Minor | 비핵심 기능 장애, 게임플레이 영향 제한적 | 다음 정기 패치 (1~2주) | UI 텍스트 잘림, NPC 대사 누락, 미니맵 표시 오류, 사운드 미재생 |
| S4 | Cosmetic | 시각적/청각적 결함, 기능 영향 없음 | 다음 메이저 업데이트 | 이펙트 잔상, 폰트 깨짐, 아이콘 미표시, 애니메이션 글리치 |

### 4.2 재현율 기반 보정

| 재현율 | 심각도 보정 |
|-------|-----------|
| 100% (항상 재현) | 등급 유지 |
| 50~99% (자주 재현) | 등급 유지 |
| 10~49% (간헐적) | 한 단계 하향 가능 (S1 제외) |
| < 10% (희귀) | 두 단계 하향 가능 (S1 제외) |

> **예외:** S1 등급은 재현율 무관 즉시 대응. 데이터 손실·보안 이슈는 1회 재현이라도 S1.

### 4.3 버그 리포트 필수 항목

```
[버그 ID]       자동 생성 (예: BUG-2026-0310-001)
[심각도]         S1 / S2 / S3 / S4
[카테고리]       전투 / UI / 네트워크 / 저장 / 인벤토리 / 퀘스트 / 성능 / 기타
[Phase]         Phase 1(웹) / Phase 2(Unity) / Phase 3(UE5)
[환경]           OS, 브라우저/엔진 버전, 해상도, 네트워크 상태
[재현 단계]      1. ... 2. ... 3. ...
[기대 결과]      ...
[실제 결과]      ...
[스크린샷/영상]  첨부
[재현율]         항상 / 자주 / 간헐적 / 희귀
[관련 파일]      소스 파일 경로 (알고 있을 시)
```

## 6. QA 인력 및 일정 가이드

### 6.1 Phase별 QA 인력

| Phase | QA 인력 | 역할 구성 |
|-------|--------|---------|
| Phase 1 (웹 MVP) | 1~2명 | QA 겸 자동화 엔지니어 1명 + 수동 테스터 1명 (파트타임 가능) |
| Phase 2 (Unity) | 2~3명 | QA 리드 1명 + 자동화 엔지니어 1명 + 수동 테스터 1명 |
| Phase 3 (UE5) | 3~4명 | QA 리드 1명 + 자동화 엔지니어 1명 + 수동 테스터 1명 + 콘솔 QA 1명 |

### 6.2 QA 일정 (Phase 1 기준 — 12주)

| 주차 | QA 활동 |
|------|--------|
| 1~2 | CI/CD 파이프라인 내 테스트 프레임워크 셋업, 유닛 테스트 템플릿 작성 |
| 3~4 | 인증/캐릭터 생성 테스트 작성, SaveLoad 무결성 테스트 구축 |
| 5~6 | 맵 이동/전투 테스트, 데미지 공식 유닛 테스트 |
| 7~8 | 전투 E2E 시나리오, 콤보/체인 시스템 테스트 |
| 9~10 | 인벤토리/퀘스트 테스트, 브라우저 호환성 테스트 시작 |
| 11 | 전체 시나리오 E2E, 성능 벤치마크, 접근성 체크리스트 |
| 12 | 릴리즈 게이트 검증, 리그레션 전수, 릴리즈 사인오프 |
