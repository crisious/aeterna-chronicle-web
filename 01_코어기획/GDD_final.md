## 1. 프로젝트 개요

### 게임 기본 정보

| 항목 | 내용 |
|------|------|
| 게임 타이틀 | 에테르나 크로니클 (Aeterna Chronicle) |
| 부제 | 망각의 신들과 마지막 증인 |
| 장르 | 실시간 반자동 전투 RPG (Semi-Auto Real-Time RPG) |
| 플랫폼 | PC 웹 브라우저 (Phase 1) / PC 클라이언트 Unity (Phase 2) / PC·콘솔 Unreal Engine 5 (Phase 3) |
| 타겟 유저 | 스토리 중심 RPG를 즐기는 PC 게이머, 20~35세, 한국 시장 1차 타겟 |
| 예상 플레이타임 | 메인 스토리 40~50시간 / 완전 클리어 80~100시간 |

### 핵심 게임플레이 루프 (3줄 요약)

플레이어는 기억 소멸이 만연한 판타지 세계 에테르나에서 기억 공명 능력을 가진 주인공 에리언을 조종하며 세계 곳곳에 흩어진 신성 기억 파편을 수집한다. 전투와 탐험을 통해 캐릭터를 성장시키는 동시에, 선택지와 서브 퀘스트를 통해 다양한 종족 세력과 동맹을 맺으며 스토리를 깊이 파고든다. 최종적으로 모든 선택이 모여 4가지 멀티 엔딩 중 하나로 귀결되며, 플레이어 행동에 따라 세계의 미래가 달라진다.

### 차별화 포인트

- **기억 메카닉의 스토리-시스템 연동**: 핵심 게임 키워드인 '기억'이 스토리, 전투 스킬(기억 소환/망각의 화살), 보스 패턴(트라우마 소환), 퀘스트 보상(기억 파편) 전반에 일관되게 녹아들어 있음
- **4가지 멀티 엔딩 + 선택지 분기**: 모든 서브 퀘스트 선택이 최종 엔딩에 영향을 미치며, 최선/정통/비극/히든 엔딩의 4분기 구조
- **웹 브라우저 RPG의 P2W 제로 설계**: "Pay for Cosmetics, Play for Power" 원칙 아래 전투 성능은 전량 게임 내 획득 가능 구조
- **스팀펑크 판타지 비주얼**: 에테르-기계 문명과 마법이 융합된 독창적 세계관으로 기존 정통 판타지 RPG와 차별화
- **웹 즉시 플레이 + Unity/Unreal 멀티 플랫폼 확장 전략**: 설치 없이 즉시 플레이 가능한 웹 버전에서 검증 후, Unity PC/모바일 및 Unreal Engine 5 콘솔 버전으로 단계적 품질 확장
- **멀티 엔진 순차 확장 전략**: Phase 1 웹에서 게임 핵심 경험을 검증한 뒤, Unity 및 Unreal Engine 5로 AAA급 품질 확장. 세 클라이언트 모두 공통 서버를 공유하는 효율적 아키텍처

## 2. 세계관 & 스토리

### 세계관 요약

**세계명: 에테르나 (Aeterna)**

에테르나는 '영원'을 뜻하지만, 역설적으로 기억이 소멸되며 서서히 사라져 가는 세계다. 세계력 3,412년, 에테르 결정(마나가 결정화된 동력원)과 증기기관이 융합된 **에테르-기계 문명**(스팀펑크 판타지) 시대를 배경으로 한다. 212년 전 (세계력 3,200년) 발생한 **대망각(The Great Forgetting)** 이후 인간과 이종족 모두 기억을 잃어가는 **기억 소멸 증후군**이 창궐하고 있다.

**주요 지역 6곳:**

| 지역 | 특징 | 분위기 |
|------|------|--------|
| 아르겐티움 | 칼리마르 제국 수도, 황금 에테르 탑 | 화려함 이면의 디스토피아 |
| 실반헤임 | 엘파리스의 기억의 숲, 기억 소멸 불가 구역 | 신비롭고 위험한 고대 숲 |
| 에레보스 | 대망각 진원지, 주민 50만 명 소멸 유령 도시 | 처연하고 고요한 공포 |
| 솔라리스 사막 | 에테르 결정 최대 매장지, 이프리타 거주 | 낮의 혹독함과 밤의 환상미 대비 |
| 북방 영원빙원 | 에테르 빙정으로 기억이 역전 복원되는 땅 | 웅장하고 숭엄한 극지 |
| 브리탈리아 자유항 | 독립 자치 무역항, 정보와 의뢰의 허브 | 활기차고 위험하며 낭만적 |

**핵심 갈등 구조:**
- **1차**: 기억 vs. 망각 — 봉인된 열두 신의 신성 기억 파편으로 인한 범세계적 기억 소멸
- **2차**: 인간 제국 vs. 이종족 연합 — 칼리마르 제국의 에테르 결정 독점과 이종족 탄압
- **3차**: 진실 vs. 편의의 거짓 — 망각의 신 레테가 인류를 조종해 다른 신들을 가둔 진짜 음모

### 세계 역사 연대표 요약

에테르나 크로니클의 역사는 태초 창세부터 게임 현재 시점(세계력 3,412년)까지
수만 년에 걸쳐 펼쳐집니다. 상세 연대표는 `시나리오/연대표_역사기록.md` 참조.

| 시대 구분 | 주요 사건 |
|-----------|-----------|
| 태초 창세 | 열두 신 출현, 에테르나 세계 창조, 레테 배제 |
| 솔리안 문명 (~세계력 기원 3,000년 전) | 에테르 문명 최전성기, 솔리안 제국 붕괴 |
| 세계력 원년 | 칼리마르 제국 건국, 인간 중심 새 문명 시대 개막 |
| 세계력 3,000년대 초 | 레테 교단 태동, 기억 소멸 징조 시작 |
| 세계력 3,200년 | 대망각(The Great Forgetting) 발생, 에레보스 50만 소멸 |
| 세계력 3,374년 | 주인공 에리언 탄생 |
| 세계력 3,412년 | 게임 현재 — 챕터 1~5, 4가지 엔딩 |

### 메인 스토리 요약

**기 (起)**: 에레보스 외곽의 떠돌이 치료사 에리언은 대규모 기억 소멸 사건을 목격하고 기억의 수호자단에 합류한다. 첫 번째 신의 봉인 해제를 위해 신성 기억 파편 수집 임무를 맡는다.

**승 (承)**: 실반헤임, 솔라리스 사막, 북방 영원빙원을 거치며 파편을 수집하는 과정에서 칼리마르 제국의 기억 사냥꾼과 레테 교단의 방해를 받는다. 에리언은 파편 수집과 함께 자신의 잃어버린 과거를 조금씩 복원한다.

**전 (轉)**: 수호자단 내부 배신자 베르나르도가 레테 교단의 첩자임이 드러난다. 에리언은 212년 전 (세계력 3,200년) 카일을 비롯한 12인의 기억 수호자가 수행한 봉인 의식의 진실 전체를 복원한다. 레테의 음모, 자신이 카일의 환생이라는 사실, 그리고 봉인을 완성하려면 자신이 희생되어야 한다는 저주.

**결 (結)**: 레테가 황제의 몸으로 전면 부상하며 에테르나가 무너지기 시작한다. 에리언은 망각의 고원 봉인석 광장에서 화신 레테와 최후의 대결을 벌이며 플레이어의 선택에 따라 4가지 엔딩으로 분기한다.

**멀티 엔딩 구조:**

| 엔딩 | 분기 조건 (요약) | 내용 |
|------|------|------|
| A: 기억의 수호자 (최선) | `fragment_count == 4` + `party_alive == 63`(전원) + `ending_a_score ≥ 60` + `lethe_understood == true` | 희생 없이 레테를 안식으로 이끌고 세계 치유 |
| B: 마지막 증인의 선택 (정통) | `fragment_count >= 3`, 위 조건 미충족 (기본 엔딩) | 에리언이 기억을 희생해 레테 봉인, 세계 구원 |
| C: 망각의 선택 (비극) | `betrayal_score >= 70` | 레테와 손잡고 선택적 망각 시스템 구축 |
| D: 신들의 귀환 (히든) | `sacred_artifacts == 12` + `sealed_ruins_visited == 7` | 열두 신 전체 해방, 속편 암시 |

> **구현 명세 참조** → [`멀티엔딩_플래그_설계.md`](멀티엔딩_플래그_설계.md)
>
> 위 조건들은 내부적으로 **플래그(Flag) 시스템**으로 관리됩니다. 핵심 플래그 목록:
>
> | 플래그 키 | 타입 | 범위 | 설명 |
> |-----------|------|------|------|
> | `story.fragment_count` | int | 0~4 | 신성 기억 파편 수집 수 |
> | `story.party_alive` | int (비트마스크) | 0~63 | 동료 6명의 생존 여부 (비트별 매핑) |
> | `story.betrayal_score` | int | 0~100 | 베르나르도·레테 공감 누적 점수 |
> | `story.artifact_count` | int | 0~12 | 신성 유물 수집 수 (메인 파편 4개 + 히든 8개) |
> | `story.seal_visited` | int (비트마스크) | 0~127 | 봉인 유적 방문 여부 (챕터별 1개, 총 7개) |
> | `story.ending_a_score` | int | 0~100 | 엔딩 A 진행도 (선택지별 포인트 누적, 임계값 60) |
> | `story.emperor_saved` | bool | — | 황제 레나르도 5세 구원 여부 |
> | `story.lethe_understood` | bool | — | 레테의 진심을 이해했는지 여부 |
>
> **엔딩 판정 우선순위**: D → C → A → B(기본값)
> 상세 선택지-플래그 매핑, DB 스키마, NPC 신뢰도 분기는 별도 문서를 참조하세요.

### 주요 캐릭터 목록

## 3. 게임 시스템

### 핵심 시스템 요약

| 시스템 | 내용 |
|--------|------|
| 최대 레벨 | 100 (+ 초월 레벨 T1~T20) |
| 클래스 체계 | 5개 기본 클래스 → 레벨 30 1차 전직 → 레벨 60 2차 전직 (총 20개 클래스) |
| 아이템 등급 | 6단계: 일반(회색) / 고급(초록) / 희귀(파랑) / 영웅(보라) / 전설(주황) / 신화(빨강) |
| 장비 슬롯 | 12 카테고리 (14 착용 포지션): 투구, 상의, 하의, 장갑, 신발, 망토, 주무기, 보조무기, 목걸이, 반지(좌/우), 귀걸이(좌/우), 장신구 |
| 화폐 체계 | 실버 / 골드 / 명성 포인트 / 레이드 토큰 / 시즌 코인 |
| P2W 원칙 | 전투 성능 관련 아이템 유료 판매 금지. 외형 및 편의성에만 유료 재화 허용 |

```

```

### 전투 방식

**실시간 반자동 전투 (Semi-Auto Real-Time with Active Pause)**

웹 브라우저 환경의 네트워크 레이턴시(50~200ms)와 마우스 전용 조작 제약을 고려해 다음 방식을 채택한다.

- **자동 기본 공격**: 조작 부담 최소화
- **수동 스킬 입력**: 1~8번 단축키로 타이밍 결정 (전략적 깊이 유지)
- **Active Pause**: 스페이스바로 일시정지 후 스킬 선택 가능
- **콤보 시스템**: 3초 이내 연속 공격 시 콤보 카운터 적립, 최대 콤보 30+ 달성 시 데미지 +50%, 크리티컬 +20%, MP 소비 -30%

**데미지 최종 계산 공식:**
```
최종 데미지 = (공격력 × 스킬 배율 - 방어력 감소 적용)
            × 레벨 차이 보정 × 속성 상성 배율
            × 콤보 보너스 × 크리티컬 배율 × 버프/디버프 배율
```

**속성 상성 (주요):**
- 화염 → 빙결/식물에 강함 (×1.5), 물에 약함 (×0.7)
- 빙결 → 바람/화염 계열에 강함, 땅에 약함
- 번개 → 물/금속에 강함, 땅에 약함
- 빛 ↔ 어둠 상호 강함

**상태이상 8종**: 독, 화상, 빙결, 기절, 출혈, 침묵, 공포, 약화

**스킬 체인 시스템**: 특정 스킬 조합을 연속 사용 시 체인 보너스 발동 (예: 아이스 랜스 후 파이어볼 → "원소 폭발", 데미지 ×2.5)

## 4. 기술 구현 계획

### 기술 스택 요약

#### 공통 서버 (세 엔진 모두 공유)
```
백엔드:       Node.js + Fastify + TypeScript + Prisma + Socket.io
데이터베이스: PostgreSQL 16 (ACID 트랜잭션) + Redis 7 (캐시/Pub-Sub)
실시간 통신:  Socket.io v4 (WebSocket + Redis Adapter)
인프라:       Docker + Kubernetes + Cloudflare CDN + AWS S3
모노레포:     pnpm + Turborepo
```

#### 클라이언트 — Phase별 엔진
```
Phase 1 (웹):     Phaser.js v3 + React + TypeScript + Vite + Zustand
Phase 2 (Unity):  Unity Engine 6 LTS + URP 2D + UI Toolkit + C# + Addressables
Phase 3 (Unreal): Unreal Engine 5.5 + UMG + GAS (Gameplay Ability System) + C++/Blueprint
```

**기술 선택 근거:**
- Phaser.js: RPG에 필요한 타일맵, 씬 관리, 물리 엔진이 내장된 2D RPG 특화 프레임워크. 60fps WebGL 렌더링 안정 달성
- Node.js + Fastify: 프론트엔드와 TypeScript 공유, 비동기 이벤트 루프로 WebSocket 최적화, Express 대비 2배 이상 처리량
- PostgreSQL: ACID 트랜잭션으로 인벤토리/거래 무결성 보장, JSONB로 유연한 아이템 속성 저장
- Redis: 세션 관리, 리더보드(Sorted Set), 서버 간 Pub/Sub 이벤트 공유

### 아키텍처 개요

#### 엔진별 플랫폼 전략

| Phase | 엔진 | 타겟 플랫폼 | 그래픽 | 설치 |
|-------|------|------------|--------|------|
| Phase 1 (MVP) | Phaser.js 웹 | PC 브라우저 | 2D 픽셀아트 | 불필요 |
| Phase 2 (알파) | Unity Engine 6 LTS | PC / 모바일 | 2D/3D 중급 | 클라이언트 설치 |
| Phase 3 (정식) | Unreal Engine 5.5 | PC 고사양 / PS5 / Xbox Series X | 3D AAA | 클라이언트 설치 |

세 클라이언트 모두 동일 서버를 공유하며, `client_type` 헤더(`web` | `unity` | `unreal`)로 서버에서 구분합니다.

#### 공통 서버 아키텍처

```
[웹 브라우저 / Unity 클라이언트 / Unreal 클라이언트]
              |
         HTTPS / WSS
              |
[Cloudflare CDN / WAF] — 에셋 캐싱, DDoS 방어, SSL
              |
[로드 밸런서 (Nginx / AWS ALB)]
    |             |             |
[API 서버]   [게임 서버]    [채팅 서버]
(Fastify)   (Socket.io)   (Socket.io)
    |             |             |
         [Redis Cluster] — 세션/캐시/Pub-Sub
              |
        [PostgreSQL 16] — Primary + Read Replicas
              |
         [AWS S3 / R2] — 게임 에셋 스토리지
```

#### 서버 설계 원칙
1. **서버 권위(Server-Authoritative)**: 모든 중요 게임 로직은 서버에서 검증하여 치팅 방지
2. **클라이언트 예측**: 입력을 즉시 로컬 반영 후 서버 응답으로 보정 (Dead Reckoning)
3. **점진적 확장**: 웹 단일 → Unity 멀티 플랫폼 → Unreal 콘솔로 단계적 확장
4. **게임 틱 레이트**: 클라이언트 60fps, 서버 20 tick/s (50ms 간격)

**성능 최적화 전략:**
- 텍스처 아틀라스로 GPU 텍스처 스왑 및 HTTP 요청 최소화 (WebP 포맷, PNG 대비 26% 절감)
- 청크 기반 맵 로딩 (32×32 타일), 가시 영역 외 오브젝트 컬링
- 오브젝트 풀링으로 몬스터 생성/소멸 비용 절감
- PostgreSQL Read Replica + Redis 캐시로 DB 부하 분산

### 웹 기반(Phase 1) 서버 환경 구성 상세

웹 플랫폼(Phaser.js)의 즉각적인 접근성과 실시간 RPG의 특성을 모두 만족시키기 위한 서버 인프라 및 환경 구성 명세입니다.

#### 1. 인프라 스택 및 아키텍처 (AWS 기반)
*   **컴퓨팅 리소스 (AWS ECS / Fargate)**:
    *   초기 트래픽 변동폭에 유연하게 대응하기 위해 컨테이너 기반(Docker) 오케스트레이션 사용.
    *   API 서버와 Game(Socket) 서버를 분리하여 독립적인 스케일링 정책 적용 (API: CPU 사용률 기반 / Game: 동시 접속자 수 기반).
*   **로드 밸런서 (AWS ALB)**:
    *   경로 기반 라우팅 지원. (`/api/*` → API 서버, `/socket.io/*` → 게임 서버)
    *   WebSocket 통신을 위한 Sticky Session 비활성화 (Redis Adapter로 상태를 공유하므로 무상태 통신 유지).
*   **CDN 및 보안 (Cloudflare)**:
    *   정적 에셋(Phaser 게임 클라이언트, 이미지, 사운드) 전역 엣지 캐싱.
    *   WAF(웹 방화벽) 및 DDoS 방어 적용하여 Socket 고갈 방지.

#### 2. 핵심 서버 프레임워크 (Node.js + Fastify)
*   **API 서버 (Fastify)**: 유저 인증(JWT), 캐릭터 정보 페칭, 상점 데이터 등 비실시간 트랜잭션 처리. (Express 대비 20% 이상 향상된 라우팅 속도)
*   **실시간 게임 서버 (Socket.io v4)**:
    *   바이너리 패킷 처리: JSON 대신 프로토콜 버퍼(Protobuf)나 MessagePack 변환을 통해 패킷 크기를 30~50% 절감하여 웹 환경의 대역폭 최적화.
    *   Room 기반 채널링: 동일 맵/던전 내의 유저만 브로드캐스트 그룹으로 묶어 패킷 전송량 최소화.

#### 3. 데이터베이스 환경 (PostgreSQL + Redis)
*   **기본 DB (Amazon RDS for PostgreSQL)**:
    *   ACID 트랜잭션 필수 구간(결제, 유저 간 거래, 장비 강화) 처리.
    *   JSONB 타입 활용으로 장비의 가변 옵션(소켓, 인챈트 수치 등) 유연하게 대응.
*   **인메모리 데이터 스토어 (Amazon ElastiCache for Redis)**:
    *   게임 서버 장애 시에도 유저 상태를 유지하기 위해 플레이어 위치, HP, 현재 상태 등을 주기적(100ms)으로 Redis에 덮어쓰기.
    *   주기적(예: 30초)으로 Redis에서 PostgreSQL로 스냅샷 Bulk 업데이트 처리 (DB 부하 방지).

#### 4. 배포 및 CI/CD 파이프라인
*   **코드 관리**: GitHub 모노레포 (Client + Server).
*   **자동화 파이프라인 (GitHub Actions)**:
    1.  PR 생성 시 TypeScript 린트(ESLint) 및 자동 테스트(Jest).
    2.  Main 머지 시 Client는 Vite 빌드 후 S3 업로드 (Cloudflare Overage 무효화).
    3.  Server는 Docker 이미지 빌드 후 Amazon ECR 푸시.
    4.  AWS ECS/Fargate 롤링 업데이트 배포 (무중단 배포).

#### 5. 로깅 및 모니터링 환경
*   **통합 로깅 (ELK/EFK Stack 또는 Datadog)**: 분산된 API와 Socket 서버의 로그를 한 곳에 수집. 트랜잭션 ID 추적으로 에러 원인 즉시 파악.
*   **APM (Application Performance Monitoring)**: Socket 지연 시간(Latency) 및 Tick 레이트 저하 알림 설정. 평균 Latency 150ms 초과 시 개발팀 슬랙 자동 알림.

### UX 원칙

**1. 신규 유저 온보딩 — 8단계 비침습적 튜토리얼**
- 별도 튜토리얼 전용 맵에서 이동 → 전투 → 인벤토리 → 장비 → 스킬 → 퀘스트 → 상점 → 오픈 월드 순서로 안내
- 모든 단계에서 "건너뛰기" 버튼 제공, 도움말 센터에서 언제든 재확인 가능
- 완료 시 실질적 보상 지급

**2. 단축키 중심 인터페이스**
- WASD + 마우스 혼용 지원, 모든 주요 창은 단축키로 즉시 접근
- 설정 화면에서 모든 단축키 재지정 가능 (충돌 시 경고)

**3. 접근성 준수**
- 텍스트 크기 75%~150% 조절, 색맹 지원 모드(적녹/청황/완전색맹) 3가지 제공
- 버튼 최소 크기 44×44px, 중요 동작(구매/버리기)은 확인 다이얼로그 필수
- 아이템 등급을 색상 외 아이콘 모양으로도 구별 (색맹 대응)

**4. 즉각적 피드백 원칙**
- 모든 버튼 클릭에 시각/청각 피드백 즉시 제공
- 자동 저장 시각적 표시(우하단 아이콘)
- 의도치 않은 아이템 버리기 5초 내 취소 가능 (토스트 메시지)

**5. 레이어 구조 (z-index)**
- 게임 월드(1) → HUD(10) → 채팅(20) → 오버레이 패널(30) → 모달(40) → 툴팁(50) → 알림(60) → 튜토리얼(70) → 로딩(80)

**6. 게임패드/컨트롤러 완전 지원 (Phase 2~3)**
- Unity: Enhanced Input System으로 PS/Xbox/Nintendo Switch Pro 지원
- Unreal: CommonUI 포커스 시스템으로 D-패드 내비게이션 완전 지원
- 컨트롤러 연결 시 HUD에 버튼 힌트 자동 전환 (키보드 ↔ 게임패드)

**7. 콘솔 플랫폼 Safe Zone 준수 (Phase 3)**
- PS5/Xbox Series X: Safe Area 5% 여백 필수 적용
- 4K(3840×2160) 기준 UI 스케일링, HDR 색상 공간 지원
- 플랫폼 트로피/업적 API 연동 팝업 알림

### 종합 대응 방안

1. **개발 리스크 완화**: Phase 1은 기술 스택의 단순한 조합(Docker Compose, 싱글플레이어)으로 시작하여 리스크를 낮춘다. 복잡한 기술(Kubernetes, 멀티플레이어 동기화)은 Phase 2에서 검증 후 도입한다.

2. **콘텐츠 리스크 완화**: 챕터 분량이 방대하므로 스토리 퀘스트는 챕터 단위로 순차 배포 전략을 채택한다. MVP에서 챕터 1 완성으로 핵심 경험을 검증하고, 유저 피드백을 반영하며 챕터 2~5를 확장한다.

3. **운영 리스크 완화**: 5가지 밸런스 KPI(유료/무료 전투력 차이, 클래스 참여 비율, 신규 유저 30일 잔존율, 일일 결제 비율, 레이드 클리어율)를 Grafana 대시보드로 실시간 모니터링하여 이상 감지 즉시 대응한다.

4. **기술 부채 관리**: 모노레포(pnpm + Turborepo) 구조와 공유 TypeScript 타입으로 프론트/백엔드 간 타입 불일치로 인한 런타임 버그를 사전 차단한다. GitHub Actions CI/CD로 자동 테스트를 의무화한다.
