## 2. 빠른 시작 (Quick Start)

> 🎯 **목표**: 5분 안에 로컬에서 게임을 실행한다.

### 사전 조건

PostgreSQL과 Redis가 로컬에서 실행 중이어야 한다. 설치가 안 되어 있으면 [3. 상세 설정](#3-상세-설정)을 먼저 참고하거나, [7. Docker 환경](#7-docker-환경)으로 건너뛰어라.

### Step 1: 저장소 클론

```bash
git clone https://github.com/crisious/aeterna-chronicle.git
cd aeterna-chronicle
```

### Step 2: 환경변수 설정

```bash
cp .env.example .env
```

기본값으로 로컬 개발이 가능하다. PostgreSQL 비밀번호만 확인:

```bash
# .env 내용 (기본값)
POSTGRES_DB=aeterna
POSTGRES_USER=aeterna
POSTGRES_PASSWORD=changeme
DATABASE_URL=postgresql://aeterna:changeme@localhost:5432/aeterna
REDIS_URL=redis://localhost:6379
PORT=3000
ALLOWED_ORIGINS=http://localhost:5173,http://localhost:80
NODE_ENV=development
VITE_SERVER_URL=http://localhost:3000
VITE_WS_URL=ws://localhost:3000
```

### Step 3: 서버 설정 + 실행

```bash
cd server
npm install
npx prisma generate
npx prisma db push        # DB 스키마 동기화
npm run dev               # → http://localhost:3000
```

### Step 4: 클라이언트 실행 (별도 터미널)

```bash
cd client
npm install
npm run dev               # → http://localhost:5173
```

### Step 5: 시드 데이터 (선택, 별도 터미널)

```bash
# 프로젝트 루트에서
npm run seed              # 마스터 시드 데이터 투입
```

### 접속 확인

| 서비스 | URL | 상태 확인 |
|--------|-----|----------|
| 게임 클라이언트 | http://localhost:5173 | 브라우저에서 열기 |
| 게임 서버 API | http://localhost:3000 | `curl http://localhost:3000/health` |
| PostgreSQL | localhost:5432 | `psql -U aeterna -d aeterna` |
| Redis | localhost:6379 | `redis-cli ping` → `PONG` |

## 3. 상세 설정

### 3.1 PostgreSQL 설정

#### macOS (Homebrew)

```bash
brew install postgresql@16
brew services start postgresql@16

# DB + 사용자 생성
createuser -s aeterna
createdb -O aeterna aeterna

# 비밀번호 설정
psql -d aeterna -c "ALTER USER aeterna PASSWORD 'changeme';"
```

#### Ubuntu/Debian

```bash
sudo apt update
sudo apt install postgresql-16

sudo -u postgres psql -c "CREATE USER aeterna WITH PASSWORD 'changeme';"
sudo -u postgres psql -c "CREATE DATABASE aeterna OWNER aeterna;"
```

#### Windows

1. [PostgreSQL 공식 사이트](https://www.postgresql.org/download/windows/)에서 설치
2. pgAdmin 또는 psql로 `aeterna` DB/사용자 생성

#### Prisma 마이그레이션

```bash
cd server

# 개발 중: 스키마 변경사항을 DB에 즉시 반영 (마이그레이션 파일 없이)
npx prisma db push

# 프로덕션: 마이그레이션 파일 기반 배포
npx prisma migrate deploy

# Prisma Client 재생성 (스키마 변경 후 필수)
npx prisma generate

# DB 초기화 (⚠️ 모든 데이터 삭제)
npx prisma db push --force-reset
```

#### DB 스키마 개요

Prisma 스키마(`server/prisma/schema.prisma`)에 정의된 주요 모델:

| 모델 | 설명 |
|------|------|
| `User` | 사용자 계정 (인증, 역할, 재화) |
| `Character` | 플레이어 캐릭터 (클래스, 레벨, 스탯) |
| `Inventory` / `Item` | 인벤토리 + 아이템 |
| `Guild` / `GuildMember` | 길드 시스템 |
| `PvpMatch` / `PvpSeason` | PvP 매칭 + 시즌 |
| `Quest` / `QuestProgress` | 퀘스트 시스템 |
| `SeasonPass` / `SeasonPassReward` | 시즌 패스 |
| `EndingFlag` / `EndingResult` | 멀티엔딩 플래그 추적 |
| `AdminAuditLog` | 어드민 감사 로그 |
| `Announcement` / `GameEvent` | 공지사항 + 이벤트 |
| 기타 | 펫, 제작, 업적, 텔레메트리 등 **총 63개 모델** |

### 3.2 Redis 설정

#### macOS (Homebrew)

```bash
brew install redis
brew services start redis

# 연결 테스트
redis-cli ping
# → PONG
```

#### Ubuntu/Debian

```bash
sudo apt install redis-server
sudo systemctl start redis-server
sudo systemctl enable redis-server
```

#### Windows

WSL2 환경에서 Linux 방식으로 설치하거나, [Memurai](https://www.memurai.com/) 사용.

#### Redis 용도

에테르나 크로니클에서 Redis는 다음 용도로 사용된다:

- **세션 캐시**: 사용자 인증 토큰
- **매칭 큐**: PvP 아레나 매칭
- **실시간 데이터**: 소켓 room 상태
- **텔레메트리 버퍼**: 이벤트 임시 저장 → PostgreSQL 동기화
- **레이트 리미팅**: API 호출 제한

> ⚠️ Redis가 없어도 서버는 구동된다 (graceful degradation). 단, 캐시와 매칭 기능이 비활성화된다.

### 3.3 환경변수 (.env) 템플릿

```bash
# ── PostgreSQL ──────────────────────────────────────────
POSTGRES_DB=aeterna
POSTGRES_USER=aeterna
POSTGRES_PASSWORD=changeme

# ── 서버 ────────────────────────────────────────────────
PORT=3000
DATABASE_URL=postgresql://aeterna:changeme@localhost:5432/aeterna
REDIS_URL=redis://localhost:6379
ALLOWED_ORIGINS=http://localhost:5173,http://localhost:80
NODE_ENV=development

# ── 클라이언트 (VITE_ 접두사 필수) ──────────────────────
VITE_SERVER_URL=http://localhost:3000
VITE_WS_URL=ws://localhost:3000

# ── 스테이징 전용 (docker-compose.staging.yml) ──────────
# STAGING_DB_PASSWORD=staging_secret
# STAGING_JWT_SECRET=staging_jwt_key

# ── 프로덕션 전용 ──────────────────────────────────────
# JWT_SECRET=<프로덕션_시크릿>
# GRAFANA_USER=admin
# GRAFANA_PASSWORD=<프로덕션_비밀번호>
```

### 3.4 시드 데이터 투입

프로젝트 루트의 `package.json`에 시드 스크립트가 정의되어 있다:

```bash
# 마스터 시드 — 전체 게임 데이터 투입
npm run seed

# 드라이런 — 실제 DB 변경 없이 시뮬레이션
npm run seed:dry

# 무결성 체크 — 시드 데이터 정합성 검증
npm run seed:check

# DB 완전 초기화 + 시드 재투입
npm run seed:reset
```

시드에 포함되는 데이터:

| 항목 | 수량 | 설명 |
|------|------|------|
| 유저 | 10명 | admin 1, moderator 2, user 7 |
| 캐릭터 | 10개 | Lv.1~80 |
| 아이템 | 80개 | 6등급 장비/소비 |
| 레시피 | 50개 | 에테르 제작 |
| NPC | 30개 | 6개 지역 |
| 퀘스트 | 60개 | 메인/서브 |
| 업적 | 100개 | 전투/탐험/소셜 |
| 이벤트 | 10개 | 운영 이벤트 |
| 펫 | 15종 + 스킬 20 | 전투/유틸 |
| 전직 | 9개 | 3클래스 × 3단계 |
| 길드 | 3개 | 각 3~5명 |

## 4. 클라이언트 빌드 (Phaser.js + TypeScript)

### 기술 스택

- **Phaser.js 3.70** — 웹 게임 엔진
- **TypeScript 5.3** — 정적 타입
- **Vite 5.1** — 빌드 도구 + HMR 개발 서버
- **Socket.io-client** — 실시간 WebSocket 통신
- **Protobuf.js** — 바이너리 직렬화 (고빈도 메시지)

### 디렉토리 구조

```
client/
├── src/
│   ├── scenes/          # Phaser 씬 (10개: Boot, Title, Game, Battle 등)
│   ├── ui/              # HUD 오버레이 (부분 렌더링)
│   ├── utils/           # ObjectPool 등 유틸리티
│   ├── telemetry/       # NPC 대화 텔레메트리
│   └── main.ts          # 엔트리포인트
├── public/              # 정적 에셋
├── package.json
├── tsconfig.json
└── vite.config.ts       # (또는 index.html 기반)
```

### 개발 서버

```bash
cd client
npm install
npm run dev
# → http://localhost:5173 (Vite HMR 활성)
```

### 프로덕션 빌드

```bash
cd client

# 타입 체크 + 빌드
npm run build

# 빌드 결과물 미리보기
npm run preview
# → http://localhost:4173

# 프로덕션 모드 빌드 (최적화 강화)
npm run build:prod
```

빌드 출력: `client/dist/` — nginx 등 정적 서버로 서빙.

### 타입 체크

```bash
cd client
npm run typecheck
# → tsc --noEmit
```

### 통신 프로토콜

클라이언트-서버 통신은 이중 프로토콜을 사용한다:

| 프로토콜 | 용도 | 파일 |
|---------|------|------|
| **Protobuf 바이너리** | 고빈도 (이동, 전투 액션) | `shared/proto/game.proto` |
| **JSON** | 저빈도 (UI, 인벤토리, 채팅) | REST API |

공유 코덱: `shared/codec/gameCodec.ts`

## 6. UE5 클라이언트 (선택)

> 고사양 PC용 데스크톱 클라이언트. 웹 클라이언트와 동일한 서버 API를 공유한다.

### 사전 조건

- **Unreal Engine 5.3+** (Epic Games Launcher에서 설치)
- **Visual Studio 2022** (Windows) 또는 **Xcode** (macOS)
- **DirectX 12** 호환 GPU

### 프로젝트 구조

```
ue5_project/
├── Config/              # 프로젝트 설정
├── Plugins/             # 커스텀 플러그인
├── Source/              # C++ 소스코드 (GAS + UMG)
├── README_UE5.md        # UE5 전용 가이드
└── README_CONSOLE_UI.md # 콘솔 UI 가이드
```

### 설정

1. Epic Games Launcher에서 UE 5.3+ 설치
2. `ue5_project/` 디렉토리에서 `.uproject` 파일을 더블클릭
3. UE5 에디터가 열리면 **Generate Visual Studio/Xcode project files**
4. IDE에서 빌드 + 실행

### 아키텍처

- **GAS (Gameplay Ability System)**: 스킬/전투 로직
- **UMG (Unreal Motion Graphics)**: HUD UI
- 서버 통신: Phaser.js 클라이언트와 동일한 Protobuf/JSON 프로토콜

> 상세한 내용은 `ue5_project/README_UE5.md` 참조.

## 7. Docker 환경

### 아키텍처

```
┌───────────────── docker-compose 네트워크 ──────────────────┐
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ Client   │  │ Server   │  │ Postgres │  │  Redis   │   │
│  │ :80      │  │ :3000    │  │ :5432    │  │ :6379    │   │
│  │ (nginx)  │──│ Fastify  │──│ v16      │  │ v7       │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  (monitoring)   │
│  │Prometheus│  │   Loki   │  │ Grafana  │                  │
│  │ :9090    │  │ :3100    │  │ :3001    │                  │
│  └──────────┘  └──────────┘  └──────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

### 풀스택 실행

```bash
# 전체 스택 (클라이언트 + 서버 + DB + Redis)
docker compose up -d

# 로그 실시간 확인
docker compose logs -f

# 상태 확인
docker compose ps

# 종료
docker compose down

# 종료 + 볼륨 삭제 (DB 초기화)
docker compose down -v
```

### 모니터링 포함 실행

```bash
# Prometheus + Loki + Grafana 포함
docker compose --profile monitoring up -d
```

| 서비스 | URL |
|--------|-----|
| 클라이언트 | http://localhost:80 |
| 서버 API | http://localhost:3000 |
| Prometheus | http://localhost:9090 |
| Grafana | http://localhost:3001 (admin / aeterna) |

### 개별 서비스 재빌드

```bash
# 서버만 재빌드
docker compose build server
docker compose up -d server

# 전체 재빌드 (캐시 무시)
docker compose build --no-cache
docker compose up -d
```

### 스테이징 환경

프로덕션과 포트가 분리된 스테이징 환경:

```bash
# 스테이징 배포 (원클릭)
bash tools/staging/deploy.sh

# 수동 실행
docker compose -f docker-compose.staging.yml --env-file .env.staging up -d

# 스테이징 DB 마이그레이션 + 시드
docker compose -f docker-compose.staging.yml exec server npx prisma migrate deploy
docker compose -f docker-compose.staging.yml exec server npx ts-node tools/staging/seed.ts

# 완전 초기화
bash tools/staging/reset.sh
```

스테이징 포트 매핑:

| 서비스 | 스테이징 | 프로덕션 |
|--------|---------|---------|
| Client | :8080 | :80 |
| Server | :3001 | :3000 |
| PostgreSQL | :5433 | :5432 |
| Redis | :6380 | :6379 |

## 8. k8s 프로덕션 배포

### 블루/그린 배포 아키텍처

```
[GitHub Tag Push] → CI (lint/test/build) → Docker Build → Green 배포
                                                            ↓
                                          헬스체크 통과 → 트래픽 전환 (Green ← 100%)
                                                            ↓
                                          Blue 유지 (롤백용)
```

### k8s 매니페스트 구조

```
k8s/
├── namespace.yaml              # etherna 네임스페이스
├── configmap.yaml              # 환경 설정
├── secrets.yaml                # 시크릿 (sealed-secrets 권장)
├── server-deployment.yaml      # 서버 Deployment
├── server-service.yaml         # 서버 Service
├── client-deployment.yaml      # 클라이언트 Deployment
├── client-service.yaml         # 클라이언트 Service
├── postgres-statefulset.yaml   # PostgreSQL StatefulSet
├── postgres-pvc.yaml           # PostgreSQL PVC
├── postgres-service.yaml       # PostgreSQL Service
├── redis-deployment.yaml       # Redis Deployment
├── redis-service.yaml          # Redis Service
├── ingress.yaml                # Ingress (트래픽 라우팅)
├── hpa.yaml                    # HPA (오토스케일링)
└── production/
    ├── server-deployment-blue.yaml    # Blue 슬롯 (대기/롤백)
    ├── server-deployment-green.yaml   # Green 슬롯 (활성)
    ├── ingress-production.yaml        # Canary 가중치 라우팅
    ├── configmap-production.yaml      # 프로덕션 설정
    ├── secrets-production.yaml        # 프로덕션 시크릿
    └── hpa-production.yaml            # 프로덕션 HPA
```

### HPA 정책

| 항목 | 값 |
|------|-----|

| 최대 레플리카 | 20 |

### 배포 명령

## 9. CI/CD

### GitHub Actions 워크플로우 (7개)

```
.github/workflows/
├── ci.yml              # PR/Push 검증 (lint → typecheck → build → docker)
├── deploy.yml          # main push → 프로덕션 배포
├── production.yml      # v* 태그 → 블루/그린 프로덕션 배포
├── staging.yml         # develop push → 스테이징 자동 배포
├── release.yml         # v* 태그 → GitHub Release 생성
├── assets.yml          # 에셋 빌드/패키징
└── db-backup.yml       # DB 백업 스케줄
```

### CI 파이프라인 (`ci.yml`)

```
Push/PR (main, develop)
  │
  ├── 1. Lint & TypeCheck
  │     ├── server: npm ci → prisma generate → tsc --noEmit
  │     └── client: npm ci → tsc --noEmit
  │
  ├── 2. Build
  │     ├── server: npm run build
  │     └── client: npm run build
  │
  └── 3. Docker Build Test
        └── docker compose build (push 없이 빌드만)
```

### 프로덕션 배포 파이프라인 (`production.yml`)

```
v*.*.* 태그 push
  │
  ├── 1. Validate — lint + tsc + unit test
  ├── 2. Build — Docker 이미지 빌드 → ghcr.io push
  ├── 3. Deploy — Green 배포 → 헬스체크 → 트래픽 전환
  └── 4. Notify — Slack 알림
```

### 릴리즈 파이프라인 (`release.yml`)

```
v* 태그 push
  │
  ├── 클라이언트 빌드 → client-dist.zip
  ├── 서버 빌드 → server-dist.zip
  ├── GitHub Release 생성
  └── 체인지로그 자동 생성
```

### 필요한 GitHub Secrets

| Secret | 용도 | 필수 |
|--------|------|------|
| `GITHUB_TOKEN` | ghcr.io push (자동 제공) | ✅ |
| `KUBE_CONFIG_PRODUCTION` | k8s 프로덕션 배포 | 프로덕션만 |
| `SLACK_DEPLOY_WEBHOOK` | 배포 알림 | 선택 |
| `DEPLOY_SSH_KEY` | SSH 배포 | 선택 |
| `STAGING_DB_PASSWORD` | 스테이징 DB | 스테이징만 |
| `STAGING_JWT_SECRET` | 스테이징 JWT | 스테이징만 |
| `TEST_DATABASE_URL` | CI 테스트 DB | 선택 |

### 패키지 권한 설정 (ghcr.io)

1. 저장소 → Settings → Actions → General
2. **Workflow permissions** → **Read and write permissions**
3. **Allow GitHub Actions to create and approve pull requests** 체크

## 11. 문제 해결 (Troubleshooting)

### PostgreSQL 연결 실패

```
Error: Can't reach database server at `localhost:5432`
```

**해결:**
```bash
# PostgreSQL 실행 상태 확인
# macOS
brew services list | grep postgresql
brew services start postgresql@16

# Ubuntu
sudo systemctl status postgresql
sudo systemctl start postgresql

# 사용자/DB 존재 확인
psql -U postgres -c "\du"     # 사용자 목록
psql -U postgres -c "\l"      # DB 목록

# .env의 DATABASE_URL 확인
echo $DATABASE_URL
```

### Redis 연결 실패

```
Error: connect ECONNREFUSED 127.0.0.1:6379
```

**해결:**
```bash
# Redis 실행 상태 확인
redis-cli ping    # → PONG이면 정상

# macOS
brew services start redis

# Ubuntu
sudo systemctl start redis-server
```

> Redis 없이도 서버는 구동됨 (graceful degradation). 캐시/매칭 기능만 비활성화.

### Prisma Client 생성 오류

```
Error: @prisma/client did not initialize yet
```

**해결:**
```bash
cd server
npx prisma generate
```

### Prisma 스키마 동기화 실패

```
Error: Prisma schema is not in sync with the database
```

**해결:**
```bash
cd server
# 개발 환경: 스키마 → DB 동기화
npx prisma db push

# 프로덕션: 마이그레이션 적용
npx prisma migrate deploy

# 최후의 수단: DB 초기화 (⚠️ 데이터 삭제)
npx prisma db push --force-reset
```

### npm install 실패

```
npm ERR! code ERESOLVE
```

**해결:**
```bash
# 의존성 충돌 시
npm install --legacy-peer-deps

# node_modules 완전 재설치
rm -rf node_modules package-lock.json
npm install
```

### TypeScript 타입 에러

```bash
# 서버 타입 체크
cd server && npm run typecheck

# 클라이언트 타입 체크
cd client && npm run typecheck

# Prisma 타입 재생성
cd server && npx prisma generate
```

### Docker 빌드 실패

```bash
# 캐시 무시 재빌드
docker compose build --no-cache

# 특정 서비스 로그 확인
docker compose logs server
docker compose logs postgres

# Docker 디스크 정리
docker system prune -af
```

### 포트 충돌

```
Error: listen EADDRINUSE: address already in use :::3000
```

**해결:**
```bash
# 포트 사용 프로세스 찾기
lsof -i :3000

# 프로세스 종료
kill -9 <PID>

# 또는 .env에서 PORT 변경
PORT=3001
```

### ghcr.io 인증 실패 (CI/CD)

```bash
# 로컬에서 ghcr.io 로그인
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin
```

### WebSocket 연결 실패

```
WebSocket connection to 'ws://localhost:3000' failed
```

**해결:**
1. 서버가 실행 중인지 확인: `curl http://localhost:3000/health`
2. `.env`의 `ALLOWED_ORIGINS`에 클라이언트 URL 포함 확인
3. `VITE_WS_URL`이 올바른지 확인

### 스테이징 ↔ 프로덕션 혼동

프로덕션과 스테이징은 **포트가 다르다**:

| | 프로덕션 | 스테이징 |
|--|---------|---------|
| Client | :80 | :8080 |
| Server | :3000 | :3001 |
| DB | :5432 | :5433 |
| Redis | :6379 | :6380 |

API 호출 시 대상 포트를 반드시 확인.
