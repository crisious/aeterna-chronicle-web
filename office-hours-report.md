## Executive Summary

Aeterna Chronicle is one of the most ambitious solo-developer game projects I've seen. 110 Prisma models, 51 route files, 15 socket handlers, 15 Phaser scenes, 1,596 hand-verified assets, 742+ design docs, 212 ending test cases, a P2W guard enforced at the code level. The craft is real.

But here's the hard question: **has anyone actually played this game?**

The path to v1.0 isn't more features. It's less distance between the game and a real human playing it.

## 1. Path to v1.0 Release

### What "v1.0" Actually Means

You have release infrastructure that most indie studios don't build until month 12:
- 9 GitHub Actions workflows (CI, staging, production, asset pipeline, db-backup)
- Docker multi-stage builds with health checks
- Kubernetes namespace configured (`etherna-production`)
- Steam deployment docs, press kit, trailer scripts, social media strategy
- Beta test plan (500 CBT + unlimited OBT via Steam Playtest)
- 62-item launch checklist covering legal, technical, marketing, accessibility

The infrastructure is ready. The game systems are complete. What's blocking v1.0 is not scope, it's **finishing the last mile**.

### The 5 Blockers (in priority order)

**Blocker 1: TypeScript Compilation is Broken**
- 165 errors across the server codebase
- This was 0 errors at Phase 48. It regressed.
- Root cause: Prisma type exports (`InputJsonValue`, `JsonNull`, `JsonValue`) + 122 implicit `any` parameters
- Impact: Production build blocked. CI pipeline fails. You can't deploy what doesn't compile.
- Fix estimate: 2-4 hours with CC. Run `npx prisma generate` first (may resolve 33 Prisma errors instantly), then add explicit type annotations to the remaining ~122 callback params.

**Blocker 2: Asset Integrity Test Failing**
- 1 asset referenced in code but missing from disk
- `tests/integration/asset-integrity.test.ts` failing
- Small but it means your verification pipeline is red. Ship green.

**Blocker 3: Launch Checklist is 0/62**
- `docs/release/launch_checklist.md` has 62 items, all unchecked
- Critical unchecked items: EULA (4 items), Privacy Policy (5 items), Age Rating (3 items)
- GRAC (Korea) and IARC (global) ratings are required before Steam/itch.io release
- This is the real calendar blocker. Legal review takes weeks, not hours.

**Blocker 4: Mobile Payments Not Implemented**
- `paymentManager.ts:40-47`: iOS returns `false`, Android returns `false`
- Apple App Store Server API v2 and Google Play Developer API are TODO stubs
- Web/Stripe works. But if you're targeting Steam, web payments via Stripe are the right first path anyway.
- Decision: Ship web-only first. Mobile can wait for Season 2.

**Blocker 5: No Load Testing**
- Beta test plan targets 500 CBT / 2,000 OBT / 5,000 spike CCU
- Zero evidence of load testing against these targets
- 15 socket handlers totaling ~69,000 lines of code handling real-time events. Under 2,000 concurrent connections with Redis pub/sub, this WILL surface bugs you haven't seen yet.
- This needs to happen before OBT. Not before CBT.

## 4. Monetization Validation

### What You've Built

The monetization design is thorough. `01_코어기획/monetization_design.md` is one of the most detailed indie game monetization docs I've read:

- **P2W Zero policy** enforced at code level (`shop/p2wGuard.ts` rejects items with stat keywords)
- **Cosmetic-only model**: skins, mounts, emotes, pet cosmetics
- **Season Pass**: 100 levels, 90-day cycle, ₩11,000-16,500 premium track
- **Crystal packages**: 6 tiers from ₩1,200 (100 EC) to ₩12,000 (12,000 EC) with first-purchase 2x bonus
- **Revenue split target**: 45% cosmetic, 25% season pass, 15% convenience, 10% direct crystal, 5% other
- **Spending tiers**: 85% free, 10% minnow (₩1,100-5,500/mo), 4% dolphin (₩11,000-33,000), 1% whale (₩55,000+)
- **Anti-pressure design**: No gacha, no limited-time FOMO, seasonal items come back 2 seasons later
- **Free player completeness**: 100% of game content accessible without paying

### The Hard Truth About Monetization

All of this is theoretical. You've designed a monetization system for a game with zero players and zero revenue data.

**What I'd challenge:**

1. **The pricing is KRW-native.** Your beta test plan targets Korea 40%, North America 30%, Japan 20%. The ₩1,200 crystal package is ~$0.85 USD. That's fine, but you need localized pricing in Stripe (USD, JPY, EUR). The `.env.example` has Stripe price IDs but they're placeholder strings.

2. **The cosmetic catalog is enormous for launch.** Phase 1 targets 9 character skins (3 per class). You have 3 base classes at launch. That's 9 skins to create pixel art for, animate, and integrate. Have you actually made any of these? The asset manifest shows character sprites but not cosmetic variants.

3. **Season Pass assumes retention.** 90-day pass, 70 days to complete. This only works if people play for 2+ months. You don't know if anyone will play for 2 hours yet. Ship the pass framework but don't activate it until you have retention data.

4. **First-purchase bonus is in the code but different from the design doc.** `paymentManager.ts` has a generic 2x `firstPurchaseBonus` flag. The design doc specifies different first-purchase mechanics. These need to reconcile.

5. **Stripe webhook is the critical path.** `stripeManager.ts` handles `checkout.session.completed` -> crystal grant. This is the one payment flow that needs to work on day one. Everything else (iOS, Android, season pass auto-renewal) can wait.

### Monetization Validation Plan

Don't validate the monetization design with spreadsheets. Validate it with behavior.

| Test | Method | When |
|------|--------|------|
| Will anyone pay ₩1,200 for 100 crystals? | Stripe test mode in CBT | CBT Week 1 |
| What do players spend crystals on first? | Analytics (`TelemetryGameEvent`) | CBT Week 2 |
| Do players hit the season pass? | Season pass progression tracking | OBT |
| P2W guard working in practice? | Monitor `[P2W-GUARD]` logs | Ongoing |
| Average session length? | Socket connect/disconnect timestamps | CBT Day 1 |

**My recommendation:** Launch with ONLY the crystal shop (6 packages) and a handful of cosmetic skins (3-5, not 27). Don't activate Season Pass until you have 30-day retention data. The P2W guard is a genuine differentiator for the Korean market where P2W backlash is fierce. Lead with it in marketing.

**Cons:**
