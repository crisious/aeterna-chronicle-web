## 테스트 흐름 (각 VU)

```
로그인 → 존 이동 → 던전 진입 → 스킬 사용(3~5회) → 아이템 획득
→ 상점 조회/구매 → PvP 매칭(매 3회차) → 소셜/리더보드(매 5회차)
→ 시즌패스(매 10회차) → 헬스체크(매 10회차)
```

## 예상 병목 지점 분석

### 1. 인증 서비스 (Critical)
- **문제**: JWT 생성 + bcrypt 비교는 CPU-intensive
- **1K VU**: 문제 없음 (p95 < 200ms 예상)
- **5K VU**: bcrypt rounds가 높으면 병목 발생 가능
- **10K VU**: rate limiter + 커넥션 풀 고갈 위험
- **대응**: bcrypt rounds 적정값 확인 (10~12), Redis 세션 캐시, 로그인 rate limit 분리

### 2. DB 커넥션 풀 (Critical)
- **문제**: PostgreSQL max_connections 기본값(100)은 5K+ VU에서 부족
- **대응**: 
  - PgBouncer 도입 (connection pooling)
  - Prisma connection_limit 조정 (`?connection_limit=50`)
  - Read replica 분리 (랭킹/리더보드 조회)

### 3. 전투 스킬 사용 (High)
- **문제**: 스킬 사용은 write-heavy (HP 계산 + 인벤토리 + 로그)
- **대응**:
  - Redis 기반 실시간 전투 상태 관리 (DB write는 배치)
  - 전투 로그 비동기 큐 처리

### 4. 거래소/상점 조회 (Medium)
- **문제**: 복잡한 필터/정렬 쿼리
- **대응**:
  - Redis 캐시 (P12-11 cacheStrategy 적용)
  - DB 인덱스 최적화 (P12-10)

### 5. PvP 매칭 큐 (Medium)
- **문제**: 동시 매칭 요청 시 Redis SORTED SET 경합
- **대응**: 매칭 서비스 수평 확장, 파티션된 큐

### 6. WebSocket 연결 (High at 10K)
- **문제**: 10K 동시 WebSocket은 단일 노드 한계
- **대응**: 
  - Socket.IO Redis adapter로 멀티노드
  - sticky session + k8s HPA

## 실행 명령어

```bash
# 기본 1000 VU
k6 run tools/loadtest/k6_scenario.js

# 5000 VU 스트레스
k6 run --env SCENARIO=medium tools/loadtest/k6_stress.js

# 10000 VU 스트레스
k6 run --env SCENARIO=heavy tools/loadtest/k6_stress.js

# 스파이크 테스트
k6 run --env SCENARIO=spike tools/loadtest/k6_stress.js

# 커스텀 서버 지정
k6 run --env BASE_URL=https://staging.aeterna.gg --env SCENARIO=medium tools/loadtest/k6_stress.js
```
