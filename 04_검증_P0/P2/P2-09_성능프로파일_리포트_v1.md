# P2-09 클라/서버 성능 프로파일 리포트 v1

**작성일:** 2026-03-11  
**상태:** 1차 최적화 적용 완료  
**대상 빌드:** 0.9.12-alpha (Phase 1 WebGL 클라이언트 + Fastify/Socket.io 서버)

---

## 1. 개요

프레임 드랍과 소켓 지연의 상한을 관리하기 위한 1차 성능 최적화.  
Phase 1 프로토타입 수준에서 "명백한 낭비"를 제거하고, Phase 2 이후 본격 최적화의 기반을 마련한다.

**핵심 원칙:**
- 매 프레임 불필요한 DOM 조작 제거
- 서버 측 방어적 rate limiting
- 리소스 누수(Redis 미정리, 타이머 미해제) 차단
- Graceful shutdown으로 데이터 유실 방지

## 3. 클라이언트 프로파일링 결과

### 3.1 렌더링 파이프라인

- **현재 구조:** Phaser Arcade Physics, 단일 스프라이트(player), 격자 배경
- **병목 없음:** 오브젝트 수가 극소 (Phase 1). draw call 1~2회 수준
- **향후 위험:** 스프라이트 증가 시 개별 텍스처 로드가 draw call 폭증 유발
- **대비:** `preload()`에 텍스처 아틀라스 로드 패턴 가이드 주석 추가

### 3.2 HUD DOM 업데이트

**Before (문제점):**
- `renderStatus()`: 매 프레임 `Math.floor(ratio * 100)` 계산 + 8개 DOM 요소 무조건 갱신
- `renderQuickSlots()`: 매 호출 시 `innerHTML = ''` → 12개 버튼 전체 재생성
- `renderQuests()`: 매 호출 시 `innerHTML` 전체 재생성
- `simulateStatusTick()`: 매 프레임(~16ms) 호출 → setStatus → renderStatus 연쇄

**After (최적화):**
- `renderStatus()`: 필드별 dirty check — 값이 변경된 DOM 요소만 업데이트
  - `lastHpPercent`, `lastMpPercent`, `lastExpPercent` 등 캐시 비교
  - 변경 없으면 DOM 접근 0회
- `renderQuickSlots()`: `dirtyQuickSlots` 플래그 — 구조 변경 시에만 full rebuild, 평상시 `updateCooldowns()`만 실행
- `renderQuests()`: `questHash` 문자열 비교 — 동일하면 innerHTML 재생성 skip
- `simulateStatusTick()`: delta accumulator로 100ms 간격 실행 (60fps 기준 ~6배 호출 감소)

**예상 효과:**
- HUD DOM 조작: ~60회/초 → ~10회/초 (상태 변경 없으면 0회)
- `renderQuickSlots` full rebuild: ~60회/초 → 슬롯 변경 시에만 (이벤트 기반)
- 퀘스트 패널: ~60회/초 → 퀘스트 데이터 변경 시에만

### 3.3 입력/이동 처리

- 클라이언트 `playerMove` emit: 200ms 스로틀 이미 적용 — 초당 최대 5패킷
- WASD/방향키 처리: Phaser input system 직접 사용, 오버헤드 무시 가능
- 숫자키 바인딩: `key.on('down')` 이벤트 방식 — 폴링 없음

### 3.4 텔레메트리 전송

- 대화 선택지 텔레메트리: 이벤트 기반 (선택 시에만 전송)
- 오프라인 폴백: `console.log`로 기록
- 패킷 크기: JSON ~500B/이벤트 — 현재 수준 문제 없음

## 4. 서버 프로파일링 결과

### 4.1 소켓 이벤트 처리

**Before:**
- `playerMove`: 클라이언트 스로틀(200ms)에만 의존. 악의적 클라이언트가 무제한 전송 가능
- `disconnect`: Redis에 저장된 `userState:{characterId}` 미삭제 → 메모리 누수

**After:**
- `playerMove`: 서버 측 50ms rate limit 추가 (Map 기반 타임스탬프 비교)
  - 정상 클라이언트(200ms)에는 영향 없음
  - 변조/버그 클라이언트의 과다 패킷 차단
- `disconnect`: `redisClient.del(userState:{socketId})` 추가 + rate limit Map 정리

### 4.2 Redis 파이프라인

- `dialogueTelemetryPersistence.ts`: `multi()` 파이프라인으로 4개 명령 원자적 실행 — 이미 최적
- TTL 60일 설정 — 자동 만료로 수동 정리 불필요
- **추가 최적화 없음** (현재 트래픽 수준에서 충분)

### 4.3 PostgreSQL 쓰기

- `persistToPostgresBestEffort`: raw SQL + ON CONFLICT UPSERT — 효율적
- 테이블 미존재 시 `console.warn` 후 서버 동작 유지 (graceful degradation)
- 현재 Phase 1에서 telemetry 외 DB 쓰기 없음

### 4.4 메모리/GC 패턴

**Before:**
- `pruneExpired()`: 매 텔레메트리 요청마다 `idempotencyStore` 전체 순회
  - Map 크기에 비례하는 O(n) 비용이 모든 요청에 부과

**After:**
- 30초 주기 `setInterval`로 분리. 타이머는 `unref()`로 프로세스 종료 차단 방지
- 요청 핫패스에서 순회 제거 → 요청 처리 지연 감소

**Graceful Shutdown 추가:**
- `SIGTERM`/`SIGINT` 핸들러: Fastify close → Redis quit → Prisma disconnect → prune 타이머 정리
- 배포/재시작 시 연결 누수 및 데이터 유실 방지

## 8. 결론

Phase 1 프로토타입의 "명백한 성능 낭비" 10건을 식별하고 수정했다.

**핵심 성과:**
1. 클라이언트 HUD DOM 조작을 83~99% 감소 — 60fps 유지에 기여
2. 서버 방어적 rate limiting 추가 — 악성 클라이언트 대비
3. Redis 메모리 누수 차단 — disconnect 시 즉시 정리
4. Graceful shutdown 추가 — 배포/재시작 안정성 확보
5. Phase 2 최적화 기반(ObjectPool, 아틀라스 가이드) 마련

현재 Phase 1 규모(플레이어 1명, 오브젝트 ~10개)에서는 성능 문제가 체감되지 않으나,  
Phase 2 멀티플레이어/파티클/이펙트 확장 시 이번 최적화가 기반이 된다.
